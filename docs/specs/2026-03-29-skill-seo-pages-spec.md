# Skill SEO Pages — Design Specification

> **Date:** 2026-03-29
> **Status:** Approved
> **Author:** zuvo:brainstorm

## Problem Statement

Zuvo has 33 skills but zero organic search presence for any of them. The zuvo.dev website has no individual skill pages — everything is either on the homepage or undiscoverable. Developers searching for "AI code quality audit," "automated security review tool," or "Playwright test generator" will never find zuvo.

Competitors (Cursor, Copilot) use monolithic feature pages that can't rank for specific skill queries. Static analysis tools (ESLint, Semgrep, SonarQube) dominate the "individual rule page" SEO space with dedicated pages per rule. Zuvo's 33 skills map perfectly to this model, and no AI coding plugin currently occupies this space.

The existing SEO audit (2026-03-28, score 53/100) identified missing sitemap, missing JSON-LD, and broken OG image as critical gates — these must be fixed alongside or before skill pages launch.

## Design Decisions

### DD1: Hybrid content generation (auto + hand-curated)

**Chosen:** Agent reads SKILL.md and generates a comprehensive YAML data file per skill. Structured data (dimensions, gates, stats) and narrative content (problem statement, FAQ, examples) coexist in one file. Agent maintains all files — no human editing.

**Alternatives considered:**
- Pure auto-generation from SKILL.md (rejected: narrative content like FAQ and problem statements need creative writing, not parsing)
- Hand-crafted pages in website repo (rejected: 33 files go stale when skills evolve, no single source of truth)

### DD2: Data files live in zuvo-plugin repo

**Chosen:** `website/skills/*.yaml` in this repo. The website repo fetches or copies them at build time.

**Rationale:** Skill definitions are authoritative here. When a skill changes, the data file updates in the same PR. Website repo is a thin rendering layer.

### DD3: One YAML file per skill, both data and narrative

**Chosen:** Single file contains structured data (stats, checks, arguments) and narrative content (problem, FAQ, examples) as markdown strings in YAML.

**Rationale:** One file per skill is simpler to maintain. No orphaned files. Agent updates the whole file atomically.

### DD4: URL structure — flat with language prefix + category hubs

**Chosen:** `/en/skills/code-audit` (individual) + `/en/skills/audits` (category hub). Language prefix from day one for future i18n.

**Alternatives considered:**
- No language prefix (rejected: migration pain when adding i18n later)
- Category in URL path `/en/skills/audit/code-audit` (rejected: deeper URLs rank worse, category is metadata not hierarchy)

### DD5: English only for v1, i18n-ready structure

**Chosen:** All content in English. URL prefix `/en/` baked in. When new languages are added, same YAML files get translated copies or `i18n` sections.

### DD6: Page template defined in spec, Astro implementation out of scope

**Chosen:** This spec defines the data schema and page template (section order, content types). The Astro implementation (components, layouts, build pipeline) is a separate task for the website repo.

## Solution Overview

```
zuvo-plugin repo                          zuvo.dev website (Astro)
┌─────────────────────┐                  ┌──────────────────────┐
│ website/skills/     │   build-time     │ src/pages/en/skills/ │
│  ├─ _schema.yaml    │ ──── fetch ────→ │  ├─ [slug].astro     │
│  ├─ code-audit.yaml │                  │  ├─ audits.astro     │
│  ├─ security-audit  │                  │  ├─ tasks.astro      │
│  ├─ brainstorm.yaml │                  │  └─ ...              │
│  └─ ... (33 files)  │                  └──────────────────────┘
└─────────────────────┘
```

An agent generates/updates the 33 YAML files by reading the corresponding SKILL.md. The website repo consumes them as a data source (Astro content collections, git submodule, or build-time fetch — implementation detail for website repo).

## Detailed Design

### Data Model — YAML Schema (v1)

```yaml
schema_version: 1
last_synced: "2026-03-29"      # date agent last regenerated from SKILL.md

meta:
  slug: string                  # kebab-case, matches skill name (e.g., "code-audit")
  title: string                 # H1 + <title> tag. Format: "{Skill Name} — {outcome phrase}"
  description: string           # <meta description>, max 155 chars, includes one quantitative claim
  category: enum                # audit | task | pipeline | utility | design | release
  og_image: string              # path: /og/skills/{slug}.png (generated by website template)

seo:
  target_queries: string[]      # 3-5 long-tail search queries this page targets (internal planning, not rendered)
  canonical: string             # /en/skills/{slug}

stats: array                    # 3-6 hero stat cards
  - label: string               # e.g., "Quality Gates"
    value: string               # e.g., "22"
    detail: string              # e.g., "CQ1-CQ22" (optional clarification)

problem: string                 # 2-4 sentence problem statement (markdown). Why this skill exists.

how_it_works: array             # 3-5 steps
  - step: string                # phase name
    description: string         # 1-2 sentences

checks: object | null           # null for non-audit skills
  sections: array               # groups of checks (gates, anti-patterns, etc.)
    - name: string              # e.g., "Quality Gates"
      id_range: string          # e.g., "CQ1-CQ22"
      count: number
      items: array              # representative items (not all — 5-8 per section)
        - id: string
          name: string
          critical: boolean

frameworks: string[]            # supported frameworks/languages. Empty for framework-agnostic skills.

modes: array | null             # null if skill has no modes/flags beyond default
  - name: string
    description: string

agents: array | null            # null if skill has no sub-agents
  - name: string                # e.g., "Code Explorer"
    role: string                # 1 sentence: what this agent does

output:
  artifact: string              # file path pattern (e.g., "audit-results/code-audit-YYYY-MM-DD.md")
  format: string                # 1 sentence: what the output looks like
  sections: string[]            # key sections in the output

examples: array                 # 1-3 examples
  - title: string
    type: enum                  # before_after | output_sample | invocation
    # -- for before_after:
    before: { language: string, code: string }
    after: { language: string, code: string }
    gate: string                # which check this demonstrates (optional)
    explanation: string
    # -- for output_sample:
    sample: { language: string, code: string }
    explanation: string
    # -- for invocation:
    command: string             # e.g., "zuvo:brainstorm build user notification system"
    result: string              # what happens (1-2 sentences)

benchmarks: array               # 1-4 verifiable claims
  - claim: string               # exact, quantitative statement
    type: enum                  # coverage | speed | accuracy | savings
    verifiable: boolean         # can the reader verify this from public SKILL.md?
    source: string              # where the number comes from

comparison_table: object | null # null for skills without meaningful comparison
  columns: string[]             # e.g., ["Capability", "Manual", "ESLint", "zuvo:code-audit"]
  rows: array of string[]
  # CONSTRAINT: cells must contain verifiable structural differences only.
  # No subjective quality claims ("better", "limited", "poor").
  # Valid: "Yes — 4 auto-FAIL gates" / "No" / "Per-rule severity"
  # Invalid: "Superior" / "Limited" / "Basic"

when_to_use: string             # 1-3 sentences: when to reach for this skill
when_not_to_use: string         # 1-3 sentences with explicit cross-links to alternative skills

related_skills: array           # 2-5 related skills
  - slug: string
    relation: string            # 1 phrase explaining the relationship

faq: array                      # 3-5 Q&A pairs targeting search queries
  - q: string
    a: string                   # 2-4 sentences. Include skill name for SEO.
    # CONSTRAINT: each Q must target a distinct search intent.
    # Mandatory intents to cover (pick 3-5):
    #   "What does X check/do?" (informational)
    #   "How is X different from Y?" (comparison)
    #   "How long does X take?" (practical)
    #   "Can X auto-fix?" (capability)
    #   "What frameworks/languages?" (compatibility)

arguments: array                # all CLI flags/arguments
  - flag: string
    description: string
```

### Schema Validation File

`website/skills/_schema.yaml` — a reference document (not executable validation) that defines the schema above. Agent uses it to self-validate when generating files.

### Page Template

The website renders each YAML into a page with these sections in order:

```
┌─────────────────────────────────────────────┐
│ BREADCRUMB: Skills > {Category} > {Name}    │
├─────────────────────────────────────────────┤
│ H1: {meta.title}                            │
│ Category badge: {meta.category}             │
│ Description: {meta.description}             │
├─────────────────────────────────────────────┤
│ STAT CARDS: {stats[]} — 3-6 cards in row    │
├─────────────────────────────────────────────┤
│ PROBLEM: H2 "The Problem"                   │
│ {problem} — 2-4 sentences                   │
├─────────────────────────────────────────────┤
│ HOW IT WORKS: H2 "How It Works"             │
│ {how_it_works[]} — numbered steps           │
│ {agents[]} — agent cards if present         │
├─────────────────────────────────────────────┤
│ CHECKS: H2 "What It Checks" (audit only)   │
│ {checks.sections[]} — grouped tables        │
│ Critical gates highlighted                  │
├─────────────────────────────────────────────┤
│ EXAMPLES: H2 "See It In Action"             │
│ {examples[]} — before/after or output       │
├─────────────────────────────────────────────┤
│ COMPARISON: H2 "Compared To Alternatives"   │
│ {comparison_table} — if present             │
│ {benchmarks[]} — verifiable claims          │
├─────────────────────────────────────────────┤
│ MODES: H2 "Modes & Options" (if present)    │
│ {modes[]} + {arguments[]}                   │
├─────────────────────────────────────────────┤
│ OUTPUT: H2 "What You Get"                   │
│ {output} — artifact path, format, sections  │
├─────────────────────────────────────────────┤
│ WHEN: H2 "When To Use It"                   │
│ {when_to_use} / {when_not_to_use}           │
├─────────────────────────────────────────────┤
│ RELATED: H2 "Related Skills"               │
│ {related_skills[]} — linked cards           │
├─────────────────────────────────────────────┤
│ FAQ: H2 "Frequently Asked Questions"        │
│ {faq[]} — accordion, FAQPage JSON-LD        │
├─────────────────────────────────────────────┤
│ CTA: Install command + docs link            │
└─────────────────────────────────────────────┘
```

**Conditional sections:**
- `CHECKS` — only for audit/design-review skills that have `checks != null`
- `AGENTS` — nested within HOW IT WORKS, only for skills with `agents != null`
- `COMPARISON` — only when `comparison_table != null`
- `MODES` — only when `modes != null`

**Sections that are always present:** BREADCRUMB, H1, STAT CARDS, PROBLEM, HOW IT WORKS, EXAMPLES, WHEN, RELATED, FAQ, CTA.

### JSON-LD Structured Data

Every skill page gets two JSON-LD blocks:

**1. BreadcrumbList** (derived from `meta.category` + `meta.slug`):
```json
{
  "@context": "https://schema.org",
  "@type": "BreadcrumbList",
  "itemListElement": [
    {"@type": "ListItem", "position": 1, "name": "Skills", "item": "https://zuvo.dev/en/skills"},
    {"@type": "ListItem", "position": 2, "name": "Audits", "item": "https://zuvo.dev/en/skills/audits"},
    {"@type": "ListItem", "position": 3, "name": "Code Audit", "item": "https://zuvo.dev/en/skills/code-audit"}
  ]
}
```

**2. FAQPage** (derived from `faq[]`):
```json
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "What does zuvo:code-audit check?",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "It evaluates every production file against 22 quality gates..."
      }
    }
  ]
}
```

The main product page (`/en/`) additionally gets `SoftwareApplication` schema.

### Integration Points

**This repo (zuvo-plugin):**
- `website/skills/*.yaml` — new directory with 34 files
- Agent reads `skills/*/SKILL.md` to generate YAML content
- No existing files modified

**Website repo (zuvo.dev, out of scope):**
- Consumes `website/skills/*.yaml` as data source
- Renders via Astro content collections or similar
- Generates OG images from template
- Produces sitemap including all `/en/skills/*` URLs
- Adds JSON-LD blocks per template logic

**Integration contract (binding on both repos):**
- YAML files are valid YAML 1.2
- Field names must not conflict with Astro content collection reserved keys (`id`, `collection`, `render`, `slug`, `body`, `data`)
- The `meta.slug` field is the unique identifier; website uses it as the URL path segment
- The website repo is responsible for the adapter layer (git submodule, copy script, or npm package — its choice)
- This repo's only obligation is producing valid, schema-conformant YAML files in `website/skills/`

### Edge Cases

**E1: Overlapping skills cannibalizing search results**

Pairs that share keyword space:
- code-audit / review → disambiguated via "batch vs per-diff" framing
- security-audit / pentest → "static posture vs exploit verification"
- write-tests / write-e2e → "unit/integration vs Playwright browser"
- design-review / design / ui-design-team → "audit existing vs create new vs multi-perspective review"
- test-audit / fix-tests → "diagnose vs repair"
- seo-audit / seo-fix → "audit findings vs apply fixes"
- ship / deploy → "pre-merge release vs post-merge production deploy"

Each skill's `when_not_to_use` field MUST name its confusable sibling and explain the difference. The FAQ MUST include a "How is X different from Y?" question for the primary confusable pair.

**E2: Pipeline skills are incomplete without each other**

Each pipeline skill page must include a visual showing the 3-step flow (brainstorm → plan → execute) with the current skill highlighted. The `when_to_use` must state the prerequisite: "Requires an approved spec from zuvo:brainstorm" (for plan) or "Requires an approved plan from zuvo:plan" (for execute).

**E2b: Release skills form a sequential pipeline**

The 5 release skills follow a natural order: ship → deploy → canary → release-docs → retro. Like pipeline skills (brainstorm → plan → execute), each release skill page should show this flow with the current skill highlighted. The `when_to_use` must reference where in the release cycle this skill fits.

**E3: Utility skills have thin content**

backlog, presentation, receive-review, worktree have few quantitative features. Strategy:
- Lead with workflow position framing ("the persistence layer for everything zuvo finds")
- Use `invocation` example type instead of `before_after`
- FAQ focuses on "when in my workflow do I use this?" queries
- Minimum 3 FAQ questions (not 5)

**E4: Comparison table claims must age well**

The `comparison_table` constraint: cells contain only verifiable structural differences. When a cell references an external tool capability (ESLint, SonarQube), it must describe what the tool structurally does, not a quality judgment. Review comparison tables quarterly.

**E5: Stats must stay synchronized with SKILL.md**

When a skill updates (e.g., CQ1-CQ22 becomes CQ1-CQ24), the YAML must be regenerated. The `last_synced` field tracks freshness. The agent should check `last_synced` vs SKILL.md modification date before any website deploy.

**E6: New, removed, or renamed skills**

When a new skill is added to the plugin, the agent creates a new YAML file in `website/skills/`. Hub pages and sitemap auto-update — no extra files needed (hubs derive from `meta.category`).

When a skill is removed, the agent deletes the YAML. The website repo should maintain a `_redirects` file (Cloudflare Pages native) mapping removed slugs to `/en/skills/` with 301 status to preserve SEO equity and avoid 404s.

When a skill is renamed (slug changes), the agent creates the new YAML and deletes the old. A 301 redirect from old slug to new slug must be added to `_redirects`.

**E7: Skills with auto-fix modes need safety callout**

Skills with `--fix-critical` or similar flags that modify files: code-audit, design-review, refactor, fix-tests. The `modes` entry for these flags must include "modifies files" in the description. The comparison table should NOT list auto-fix as a pure advantage without the scope caveat.

## Acceptance Criteria

### Must-have

1. 39 YAML files in `website/skills/`, one per skill, conforming to the schema
2. Every file has a unique `meta.title` containing the skill name and one quantitative claim
3. Every file has `meta.description` under 155 characters with at least one number
4. Every file has `stats[]` with 3-6 verifiable stat cards
5. Every file has `problem` statement (2-4 sentences)
6. Every file has `how_it_works` with 3-5 steps
7. Every file has `examples[]` with at least 1 example
8. Every file has `when_to_use` and `when_not_to_use` with cross-links to sibling skills
9. Every file has `related_skills[]` with 2-5 entries
10. Every file has `faq[]` with 3-5 Q&A pairs targeting distinct search intents
11. Every confusable skill pair has a disambiguation FAQ question on both sides
12. Every pipeline skill references its prerequisite/output artifact
13. `_schema.yaml` reference document exists
14. `comparison_table` cells contain only verifiable structural differences (no subjective claims)
15. Hub pages are derivable: every skill YAML has `meta.category` set correctly so the website can generate the 5 hub pages + index page automatically by grouping on this field. No additional hub data files needed in this repo.

### Should-have

15. Audit skills have `checks` section with representative items from each gate group
16. Pipeline skills have `agents[]` with named roles
17. Skills with auto-fix modes have safety callout in `modes[]` description
18. `benchmarks[]` present on audit and task skills (at least 1 claim each)
19. `frameworks[]` populated for all framework-aware skills
20. `seo.target_queries` populated with 3-5 long-tail queries per skill

### Edge case handling

21. Utility skills use workflow-position framing in `problem` and `when_to_use`
22. Utility skills use `invocation` example type, not `before_after`
23. Pipeline skill pages reference the 3-step flow with prerequisite state
24. `last_synced` date matches the date the file was generated

## Out of Scope

- **Astro implementation** — components, layouts, build pipeline, content collections are website repo concerns
- **OG image generation** — template and tooling for generating 33 PNGs from skill data
- **Sitemap generation** — Astro plugin handles this
- **Existing SEO fixes** — the 4 critical issues from the 2026-03-28 audit (robots.txt, sitemap, OG image, JSON-LD) are a prerequisite but a separate task
- **i18n content** — translated YAML files for non-English languages
- **Analytics/tracking** — conversion tracking, click analytics on skill pages
- **Interactive demos** — live playground or embedded skill execution
- **Blog content** — supporting blog posts per skill for content marketing
- **Versioned skill pages** — historical versions of skill content

## Open Questions

None — all design decisions resolved in the dialogue. The website repo's choice of integration mechanism (git submodule, build-time fetch, or copy script) is intentionally left to that repo. The integration contract above ensures the YAML files are consumable regardless of which mechanism is chosen.
