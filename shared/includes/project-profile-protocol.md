# Project Profile Protocol

> Load the project's convention profile from `.zuvo/project-profile.json`. Falls back to legacy inline detection when unavailable.

## Purpose

Replace per-skill inline stack detection with a shared, deterministic project profile generated by CodeSift's `analyze_project()` tool. The profile contains:

- **identity** — project name, type, workspace root, git remote
- **stack** ��� framework, language, test runner, package manager, monorepo
- **file_classifications** — critical/important/routine tiered
- **dependency_graph** — entry points, hub/leaf modules, orphans
- **test_conventions** — mock patterns, setup files, common mocks, assertion style
- **known_gotchas** — auto-detected project-specific pitfalls
- **conventions** — framework-specific (Hono/NestJS/Next.js/Express/React/Python/PHP)
- **dependency_health** — total deps, key versions
- **git_health** — commits, velocity, contributors

## Schema Version

This protocol loads profiles conforming to **schema v1.0** (`compatible_with: ">=1.0, <2.0"`).

## Full Protocol (12 steps)

### Step 0: Rollback check

Check `ZUVO_USE_PROFILE` env var. If `"false"` → skip everything, use legacy. Return immediately.

### Step 1: Lock

Acquire `.zuvo/.profile.lock` (timeout 30s). If lock exists, check PID — if stale (dead process), remove and acquire. If lock held by live process, wait up to 30s then proceed without lock.

### Step 2: Read profile

Try to read `.zuvo/project-profile.json`.

### Step 3: Parse check

If JSON parse fails → log to `.zuvo/profile-errors.log` with timestamp → treat as missing → go to Step 7.

### Step 4: Sanity check

Verify loaded profile:
- Has `version`, `status`, `generation_metadata` fields
- `generation_metadata.files_analyzed > 0`
- At least one content section exists (`stack`, `file_classifications`, or conventions)

If sanity fails → log → treat as missing → go to Step 7.

### Step 5: Version compatibility

Check `compatible_with` field. If current protocol version is outside range → go to Step 7 (regenerate).

### Step 6: Cache invalidation (5 triggers, 50ms budget)

Check staleness in priority order. Stop checking if 50ms budget exhausted (accept as fresh):

1. **Critical files** — mtime of `package.json`, `tsconfig.json`, `vitest.config.*`, main entry point > `generated_at`
2. **Middleware dirs** — mtime of any `*/middleware/` directory > `generated_at`
3. **Recent tests** — mtime of 5 most recent `*.test.*` files > `generated_at`
4. **User overrides** — mtime of `.zuvo/profile-overrides.json` > `generated_at`
5. **Extractor version** — `generated_by.extractor_versions` differs from CodeSift's `get_extractor_versions()`

If any trigger fires → go to Step 7.
If `--no-cache` flag → go to Step 7.
If all pass → go to Step 9.

### Step 7: Generate

Check CodeSift availability.

**If available:** call `analyze_project(repo_id, force=true)` → profile saved to disk automatically → go to Step 9.

**If unavailable:** go to Step 8.

### Step 8: Legacy fallback

Use inline stack detection (read `package.json`, `tsconfig.json`, etc.). Set `degraded` flag.
Print once: `[PROFILE] CodeSift unavailable — using legacy stack detection.`
Go to Step 12.

### Step 9: Merge overrides

If `.zuvo/profile-overrides.json` exists:
- Deep merge into loaded profile
- Scalars: override replaces base
- Objects: recursive merge
- Arrays: override appends to base
- `null` value: deletes key from base
- Merged result is NOT written back to disk

### Step 10: Status check

- `status: "complete"` → go to Step 11
- `status: "partial"` → set `degraded` flag, go to Step 11
- `status: "failed"` → go to Step 8

### Step 11: Return profile

Profile is ready. Go to Step 12.

### Step 12: Release lock

Release `.zuvo/.profile.lock`. Return profile (or degraded flag) to skill.

## Profile Consumption by Skills

**MANDATORY for all skills that analyze or modify code.** Do NOT skip regardless of file type.

```
ALWAYS USE (every file type):
  → profile.stack — framework, test runner, language
  → profile.test_conventions — mock patterns, setup files, common mocks
  → profile.file_classifications — know if target is critical/important/routine
  → profile.identity — project name, type
  → profile.dependency_graph — hub/leaf context for the file

FRAMEWORK-SPECIFIC (when profile has conventions):
  → ORCHESTRATOR: middleware_chains, rate_limits, route_mounts, auth_patterns
  → CONTROLLER/SERVICE: global guards, modules, middleware context
  → GUARD/MIDDLEWARE: position in global chain, what it protects
  → COMPONENT/HOOK: state management, UI library, routing
  → ALL: test_conventions.mock_patterns for project-specific mock setup
```

Print confirmation: `[PROFILE] Loaded: {framework} + {test_runner}, {mock_style}, {N} critical files`

If profile not available: `[PROFILE] Not found — using legacy detection.`

## `.gitignore` Verification

Before writing `.zuvo/project-profile.json`, verify `.zuvo/` is in `.gitignore`. If absent, append:
```
# zuvo session state (local runtime, not committed)
.zuvo/
```

## `--no-cache` Flag

Skills accepting `--no-cache` skip cache check in Step 6 and force regeneration. Maps to `analyze_project(force=True)`.
