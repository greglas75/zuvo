# Project Profile Protocol

> Load the project's convention profile from `.zuvo/project-profile.json`. Falls back to legacy inline detection when unavailable.

## Purpose

Replace per-skill inline stack detection with a shared, deterministic project profile generated by CodeSift's `analyze_project()` tool. The profile contains stack info, file classifications, framework conventions, and test conventions — all with `file:line` evidence.

## Schema Version

This protocol loads profiles conforming to **schema v1.0** (`compatible_with: ">=1.0, <2.0"`).

---

## Phase 1A Protocol (current)

### Step 0: Environment check

Check the `ZUVO_USE_PROFILE` environment variable. If set to `"false"`, skip the entire protocol — fall to Step 3 immediately. This is the global rollback mechanism.

### Step 1: Read cached profile

Try to read `.zuvo/project-profile.json`.

**If missing or JSON parse error:** proceed to Step 2.

**If valid JSON:**

1. Check `generated_at` timestamp against mtime of critical files:
   - `package.json`
   - Main entry point (detect from profile's `file_classifications.critical` or `app.ts`/`main.ts`/`server.ts`/`index.ts` at project root)
2. If any critical file has mtime **newer** than `generated_at` → profile is stale → proceed to Step 2.
3. If profile is fresh → return profile to skill. Done.

### Step 2: Generate profile

Check whether CodeSift MCP tools are available (per `codesift-setup.md`).

**If CodeSift available:**

1. Call `analyze_project(repo_id)` — returns JSON conforming to schema v1.0.
2. Write the result to `.zuvo/project-profile.json`.
3. Return profile to skill. Done.

**If CodeSift unavailable:** proceed to Step 3.

### Step 3: Legacy fallback

Fall back to the skill's existing inline stack detection (read `package.json`, `tsconfig.json`, etc.). Set a `degraded` flag so the skill knows conventions are not available.

Print (once per session): `[PROFILE] CodeSift unavailable — using legacy stack detection. Install CodeSift for convention-aware test generation.`

---

## Profile Consumption by Skills

When a skill receives the profile:

```
IF profile.status == "complete" AND profile.conventions exists:
  → Use convention values for framework-specific behavior
  → For ORCHESTRATOR tests: use middleware_chains, rate_limits, route_mounts, auth_patterns
  → For other code types: use test_conventions for mock style, assertion patterns

IF profile.status == "partial":
  → Use available sections, fall back to generic patterns for missing sections

IF profile.status == "failed" OR degraded flag set:
  → Use legacy generic patterns (current behavior)
```

## `--no-cache` Flag

Skills that accept `--no-cache` skip Step 1 entirely and go directly to Step 2 (regenerate). This maps to `analyze_project(force=True)` on the CodeSift side.

Example: `zuvo:write-tests app.ts --no-cache`

## `.gitignore` Verification

Before writing `.zuvo/project-profile.json`, verify that `.zuvo/` appears in the project's `.gitignore`. The profile contains absolute paths (`identity.workspace_root`) and is machine-local — it must not be committed.

If `.zuvo/` is not in `.gitignore`, append:
```
# zuvo session state (local runtime, not committed)
.zuvo/
```

This check is also performed by `zuvo:execute` during session initialization (see `session-state.md`).

## Phase 1B Extensions (not yet implemented)

The following will be added when Phase 1B ships:

- **Lock file** (`.zuvo/.profile.lock`) for concurrent skill execution
- **Sanity checks** on loaded profile (required sections exist, metadata sensible)
- **Corruption recovery** (parse failure → log to `.zuvo/profile-errors.log` → regenerate)
- **Partial status handling** (profile with `status: "partial"` returned with degraded flag)
- **Version compatibility check** (`compatible_with` field vs protocol version)
- **5-trigger cache invalidation** with 50ms budget (critical files, middleware dirs, recent tests, user overrides, extractor versions)
- **Profile overrides merge** (`.zuvo/profile-overrides.json` deep-merged into base profile)
- **Migration metrics logging**
